Introduction

In a comparison of the accessibility of web pages, a page belonging to __org__ received a deficit score of __totalScore__. This report explains how that score was computed.

The pages were tested with version __version__ of the __scoreProc__ procedure of Autotest. A total of 427 tests were performed on each page. The resulting data were saved in a JSON-format file.

Test packages

Most of the tests belonged to accessibility test packages created by other specialists.

Axe

__axeResult__

Axe is an open-source package sponsored by accessibility consulting firm Deque. The procedure performs all 138 of the default Axe tests.

Equal Access

__ibmResult__

Equal Access is an open-source package sponsored by IBM Corporation. The procedure performs all 163 of its default tests.

WAVE

__waveResult__

WAVE is a proprietary package owned by webAIM, a program of the Institute for Disability Research, Policy, and Practice at Utah State University. The procedure performs all 110 of its default tests.

Custom tests

The tests in the above packages are designed to detect some, not all, accessibility problems. I have created more tests and included them in this procedure to detect some problems not detected by the test packages.

bulk

__bulkResult__

The bulk test counts the initially visible elements in a page. A page with a large count tends to be complex and busy, frustrating some users, especially if they have visual or motor disabilities, as they try to determine what the page is about, whether it is relevant, and how to find a specific thing in it.

When the count exceeds 250, the procedure begins to assign a non-zero deficit score.

embAc

__embAcResult__

The embAc test detects improper embedding of interactive elements (links, buttons, inputs, and select lists) within links or buttons. Such embedding violates the HTML standard, complicates user interaction, and creates risks of error. It becomes non-obvious what a user will activate with a click.

focAll

__focAllResult__

The focAll test detects discrepancies between focusable and Tab-focused element counts. Navigating with the Tab key normally moves the focus and does nothing else. If it also adds or fails to focus focusable elements, this complicates navigation and may make the page unusable for people who must use only a keyboard (not a mouse) or a keyboard-emulating assistive device to navigate.

Testing failures

Some pages prevent some of the tests in this procedure from being performed. This may occur, for example, when a page tries to block any visitor that is not a real human being. The procedure assigns deficit scores when pages prevent tests. This treatment indicates that preventing accessibility testing is itself an accessibility deficiency, because:

  • Measuring success is a prerequisite for achieving success, so interfering with accessibility measurement interferes with accessibility.
  • Users with disabilities often rely on assistive technologies to mediate between them and web applications. Measures that interfere with automated testing are at risk of interfering with some assistive technologies, too.
