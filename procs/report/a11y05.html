Custom tests

Although these three packages conduct a total of 411 tests, the tests are designed to detect only some, not all, accessibility problems. In general, these packages inspect the code of a web page, but they do not observe changes in a page, navigate within a page, or perform operations on a page. Some accessibility problems arise only over time, or only when a user performs an action (such as pressing a key, hovering, or clicking). I have created more tests and included them in this procedure to detect some problems not detected by the test packages.

bulk

The bulk test counts the initially visible elements in a page. The motivation for this test is the frustration that users with visual and motor disabilities (and not only they) often express with content bloat. A complex, busy page usually makes it difficult to know what the page is about, whether it is relevant, and how to find a specific thing there. This procedure begins to assign a deficit when the count exceeds 250. The count for this page was __bulkCount__. Obviously, it requires judgment, and perhaps user research, to determine whether this reflects practical accessibility.

embAc

The embAc test detects improper embedding of interactive elements (links, buttons, inputs, and select lists) within links or buttons. Such embedding not only violates the HTML standard, but also complicates user interaction and creates risks of error. It becomes non-obvious what a user will activate with a click.

focAll

The focAll test detects discrepancies between focusable and Tab-focused element counts. The test first counts all the visible Tab-focusable (i.e. with tabIndex 0) elements (except counting each group of radio buttons as only one focusable element). In principle, by pressing the Tab key a user should be able to reach (i.e. focus) each of those elements. Then the test repeatedly presses the Tab (or Option-Tab in the webkit browser) key until it has focused all the elements it can and counts those elements. If the latter are more than the former, Tab-key navigation made more focusable elements visible. If the latter are fewer than the former, the page managed focus so as to prevent Tab-key navigation from focusing all the elements it would normally be able to focus. Either of these can complicate navigation for users. It may disappoint the expectation that the content will remain stable as they move with the Tab key, or the expectation that they can reach every focusable element (or widget, such as one radio button or tab in each group) merely by pressing the Tab key. This problem is acute for users who must use only a keyboard (not a mouse) or a keyboard-emulating assistive device to navigate.
